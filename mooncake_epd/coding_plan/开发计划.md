## 0. 结论摘要

当前代码已经具备若干单机原型模块：页级 KV 管理、RadixTree 前缀缓存、FeatureStore、TransferPolicy、本地 TransferEngine、EPD worker、Workflow、调度与 offload 原型等。但整体仍停留在“单进程 / 本地 tensor copy / demo 验证”阶段，和 RFC 要求的“统一多模态状态对象 + EPD 三阶段分离 + 分布式 owner shard 控制面 + Agent 跨步骤复用 + A2A / Offload / 可靠性压测”存在显著缺口。

必须优先修复的 P0 问题：

1. **状态对象不符合 RFC v2 语义**：`MultimodalState` 缺少 `version_id`、`parent_version_id`、`snapshot_epoch`、`status`、`approximate`、全局 `BlockRef` 字段等，无法提供 workflow 内 snapshot isolation。
2. **引用计数只有本地 page manager 级别**：`BlockRef` 只有 `physical_id/filled`，没有 `global_block_id`、owner shard、逻辑位置、virtual offset；无法支持跨节点 fork、A2A、RDMA/TCP/SHM 统一 ownership。
3. **RadixTree scope 未实现**：`match_longest_prefix(scope=...)` 明确忽略 scope，可能跨 workflow 错复用 KV。
4. **FeatureStore 没有真实 refcount**：`incref` 只是刷新 LRU，`release` 是 no-op，不满足 RFC 的 FeatureBundle 生命周期要求。
5. **TransferEngine 跨节点路径未实现**：Mooncake 初始化后 `_remote_transfer` 仍 `NotImplementedError`，P→D/E→P/A2A/Offload 只有本地复制或 CPU round-trip 模拟。
6. **跨步骤复用缺少安全闸门**：当前 RelayRecompute 近似实现没有 Virtual-RoPE、KV Validity Check、attention drift estimator、fallback-safe 质量闸门。
7. **Offload 不是 RFC 的异步状态机**：当前 offload 同步复制 CPU、直接释放 GPU refs，无 `OFFLOADING/RESTORING` lock、ABORT_DMA、in-flight restore、MooncakeStore 语义。
8. **测试体系不合格**：`compileall` 通过；但 pytest 下 24 passed / 8 errors，Phase1/Phase3 的 real model 测试被 pytest 收集却缺 fixture，无法作为 CI gate。

本计划建议先完成语义地基与测试闸门，再推进真实 EPD / Mooncake / vLLM 集成，最后做 Agent 调度、A2A、RDMA 与评估矩阵。

## 1. 审查证据与现状问题

### 1.1 状态层问题

相关文件：

- `core/state/state.py`
- `core/state/page_manager.py`
- `core/state/radix_tree.py`
- `core/state/feature_store.py`
- `agent/coordination/workflow.py`

发现：

- `StateMeta` 只有 `token_ids/image_ids/agent_id/workflow_id/step/turn/ttl_deadline/priority`，缺少 RFC §4.1 的 `version_id/parent_version_id/snapshot_epoch/approximate/status`。
- `MultimodalState` 只有本地 `state_id/kv_refs/feature_hash/feature_hashes/meta`，没有不可变版本语义；fork 复制 meta 后直接登记 child，无法保证父状态并发 decode/fork 时的 snapshot isolation。
- `BlockRef` 只有 `physical_id` 和 `filled`，无法表达 RFC 要求的 `global_block_id/physical_node_id/logical_index/virtual_offset`。
- `PagedKVManager` 使用进程内 `_pages` 和 sharded lock，适合单机原型，但不具备 owner shard 权威、global GID、跨节点 CAS 提权、物理释放确认等能力。
- `StateLayer.handoff()` 只是改 `agent_id`，没有 2PC prepare/commit、目标节点 pull、失败回滚。
- `StateLayer.offload()` 直接 `release_refs` 并清空 `state.kv_refs`，对极速 restore、并发 fork、restore-after-fork race 没有锁语义。

### 1.2 前缀缓存 / 跨步骤复用问题

相关文件：

- `core/state/radix_tree.py`
- `core/state/relay_recompute.py`
- `agent/coordination/reuse_pipeline.py`
- `agent/coordination/workflow.py`

发现：

- `RadixTree.match_longest_prefix(..., scope=...)` 文档说明 scope 参数“accepted and ignored”，即 workflow 隔离没有实现。
- `RadixTree` 节点没有 workflow_id、epoch、version 或 owner metadata，无法实现 per-workflow snapshot 查询。
- `Workflow.advance()` 通过 `prev_tokens + new_tokens` 做 prefix 查询，但没有基于 `snapshot_epoch` 查询稳定版本。
- `RelayRecompute` 的 `residual_attention_merge()` 只是拼接 K/V，并没有真实 residual attention kernel、Virtual-RoPE、mask/position correctness 检查。
- `ReusePipeline` tier3 当前说明“Currently a no-op”，但测试描述把它当作完整 tier3 能力，存在能力声明与实现不一致。

### 1.3 FeatureBundle 缓存问题

相关文件：`core/state/feature_store.py`

发现：

- `hash_pixel_values()` 对大 tensor 采样哈希，RFC §6.9 要求 URL/path → 元数据 → 全量/异步三段式极速哈希，当前未实现 URL/path 和 metadata hash。
- `FeatureStore.incref()` 只是刷新 LRU；`release()` 是 no-op。RFC 要求 FeatureBundle 与 KV pages 一样具备引用计数与 owner shard 生命周期。
- FeatureBundle 无 owner、lease、ttl_deadline、refcount、状态锁，无法跨节点共享与 A2A 迁移。

### 1.4 传输层问题

相关文件：

- `core/transfer/policy.py`
- `core/transfer/engine.py`
- `core/transfer_engine.py`

发现：

- `TransferPolicy` 已有 mode/compress/precision/prefetch 枚举，但缺 RFC §5.5 的 cost model、adaptive controller、候选策略画像和 EWMA 回归。
- `TransferEngine.transfer_tensor()` 在 local 下执行 tensor copy；CacheGen 通过 CPU round-trip 模拟。
- `TransferEngine.initialize()` 可初始化 Mooncake，但 `_remote_transfer()` 在 Mooncake 存在时直接 `NotImplementedError`，真实 TCP/RDMA path 未落地。
- `transfer_pages()` 创建新的本地 `PagedKVManager`，没有 owner shard incref、borrow/own 标记、RDMA MR、Mooncake Store handle。
- `transfer_async()` 只是线程池调用 `_dispatch_sync()`，没有按 E2P/P2D/A2A/Offload channel 做流式、pull、push_batch 或 prefetch。

### 1.5 EPD workers / vLLM 集成问题

相关文件：

- `core/epd_workers.py`
- `core/epd_pipeline.py`
- `demo/vllm_integration.py`

发现：

- `core/epd_workers.py` 尝试接入 Qwen3-VL real model，但 Prefill 对 DeepStack 的调用依赖具体 HF 内部接口，缺少兼容性探测、版本锁定和 fallback。
- `DecodeWorker.decode()` 只 materialize 全量 KV 到 DynamicCache，再用 HF model 继续 decode；这不是 vLLM PagedAttention block 原地消费，也没有 CoW 写回新增页。
- 旧版 `core/epd_pipeline.py` 仍使用 `encoder_worker.py/prefill_worker.py/decode_worker.py` 的 mock/简化接口，与新版 `core/epd_workers.py` 并存，抽象重复且容易漂移。
- vLLM MooncakeConnector 相关配置生成存在 demo，但没有端到端启动、健康检查、请求路由与真实 P→D 验证。

### 1.6 调度 / Offload / A2A 问题

相关文件：

- `agent/coordination/scheduler.py`
- `agent/coordination/elastic_ratio.py`
- `agent/coordination/offload.py`
- `core/state/anchor_pool.py`

发现：

- Scheduler 是本地启发式 score，没有 RFC §6.14 的 M/M/k 队列模型、rho 阈值、反压、admission control、deadline P99 预测。
- Preemption 当前主要管理 request 对象；没有和真实 decode slot、MooncakeStore、StateLayer lock 串联。
- Offload 同步复制 CPU，缺少非阻塞 DMA、MooncakeStore、状态机、ABORT_DMA、in-place reclamation。
- A2A AnchorPool 是本地 prefix anchor 查找，不等价于 2PC handoff + increment pull + rollback。

## 2. 总体目标与验收标准

### 2.1 工程目标

实现 RFC 的最小可验收 Full System 路径：

1. EPD 三阶段：Encoder → Prefill → Decode 可本地多 GPU 跑通，P→D 能接 vLLM/MooncakeConnector 或明确的 Mooncake transfer adapter。
2. 统一状态对象：版本化 `MultimodalState`、全局 `BlockRef`、页级 refcount/CoW、FeatureBundle refcount、workflow scoped RadixTree。
3. Agent 能力：fork、advance_step、handoff、offload/restore、priority scheduling 具有可测试语义。
4. 跨步骤复用：exact prefix 默认开启；Virtual-RoPE / approximate reuse 在安全闸门与 full recompute fallback 下逐步开启。
5. 可靠性：refcount mismatch、UAF、double-free、orphan block、deadlock 等指标自动化断言。
6. 评估：覆盖 RFC W0-W5 workload 的最小 synthetic/dev-small 版本，报告 TTFT/JCT/reuse/fork/offload/quality/failure 指标。

### 2.2 分阶段验收线

- **P0 通过线**：单机 CPU/CUDA 单元测试全部 pytest 化；状态层版本语义、scope 隔离、FeatureStore refcount、refcount 不泄漏。
- **P1 通过线**：本地多 GPU EPD 跑通；P→D transfer 可替换；测试能比较 single-node baseline vs local EPD。
- **P2 通过线**：Agent fork/advance_step/offload/handoff 的 synthetic 压测通过，UAF/double-free/refcount mismatch 为 0。
- **P3 通过线**：Mooncake/vLLM connector 真实链路打通；没有 RDMA 时 TCP 可跑；RDMA 不可用不影响 correctness。
- **P4 通过线**：W0-W5 dev-small 完成指标报告，满足 RFC §8.9 的基本质量与稳定性约束。

## 阶段 B：状态层语义地基重构（P0/P1，预计 3-5 天）

### B1. 实现 RFC v2 版本化状态对象

任务：

- 扩展 `BlockRef`：
  - `global_block_id: str`
  - `physical_node_id: str`
  - `logical_index: int`
  - `virtual_offset: int = 0`
  - 保留本地 `physical_id` 作为 local backend 字段。
- 扩展 `MultimodalState`：
  - `workflow_id`
  - `step_index`
  - `version_id`
  - `parent_version_id`
  - `snapshot_epoch`
  - `approximate`
  - `status: ACTIVE|OFFLOADING|RESTORING|HANDING_OVER|ABORTED`
  - `ttl_deadline`
- 将 `StateMeta` 中的 workflow/step/ttl 字段逐步提升为 state 顶层字段，保留 `meta` 存 agent/user metadata。
- `fork()` 必须继承父 `snapshot_epoch`，设置 `parent_version_id=parent.version_id`，子版本不可变。
- `advance_step_commit()` 必须创建新 `snapshot_epoch + 1` 或通过 `EpochManager.commit_step()` 生成。

验收：

- 新增 `tests/test_state_versioning.py`：
  - fork 后父状态 advance/decode 不改变子状态 token/kvs。
  - old version release 不影响 child version。
  - snapshot_epoch 单调递增。

### B2. 引入本地 OwnerShard / KV Directory 抽象

任务：

- 新增 `core/control/kv_directory.py`：本地内存版 owner shard，API：
  - `register_block(local_pid, workflow_id, node_id) -> global_block_id`
  - `incref(gid)` / `decref(gid)`
  - `promote_block(gid, node_id) -> PromoteResult`
  - `lease(gid)` / `release_physical(gid)`
  - `commit_epoch(workflow_id, version_id)`
- `PagedKVManager` 不再作为唯一 refcount 权威；本地模式可用 adapter 把 `gid -> physical_id` 映射回 `_pages`。
- 先做 single-process owner shard，保留未来 per-shard Raft/Mooncake backend 插口。

验收：

- fork/refcount/cow 的权威数据来自 KV Directory。
- 128 branch synthetic CoW race：refcount mismatch = 0，UAF = 0，double-free = 0。

### B3. 修复 FeatureStore 生命周期

任务：

- 为 FeatureBundle 引入 `FeatureRef`：`feature_bundle_id/global_id/owner_shard/node_id/refcount/ttl/status`。
- `FeatureStore.incref/release` 实现真实 refcount；refcount > 0 时 LRU 不得删除。
- 实现 RFC §6.9 三段式 hash：
  - L1 URL/path string hash。
  - L2 shape + dtype + resolution + first/last 1KB bytes。
  - L3 full hash async，可更新弱 hash alias。

验收：

- 多 state 共享同一 FeatureBundle，释放任一 state 不删除仍被引用 bundle。
- LRU 只淘汰 refcount=0 的 FeatureBundle。

### B4. 实现 workflow scoped RadixTree

任务：

- RadixTree node 增加 `scope/workflow_id/snapshot_epoch/version_id` metadata。
- `insert(token_ids, refs, scope, epoch, version_id)`。
- `match_longest_prefix(token_ids, scope, epoch)` 只匹配同 workflow 或明确允许的 stable shared namespace。
- 防止不同 workflow 相同 token 但不同图像/不同 feature context 错复用。

验收：

- 新增测试：两个 workflow token 完全相同但 image hash 不同，不得错配 KV。
- `scope` 参数不再被忽略。

## 阶段 C：安全跨步骤复用（P1/P2，预计 4-6 天）

### C1. Exact prefix reuse 正确化

任务：

- `StateLayer.advance_step_prepare()` 改为基于 `workflow_id + snapshot_epoch` 查询。
- exact prefix 返回 refs 时由 owner shard lease/incref，commit 后清理临时 lease。
- 统计 `matched_tokens/matched_pages/saved_prefill_flops`。

验收：

- tail append 场景 bit-exact：启用 exact prefix reuse 与 full prefill logits/KV 对齐。
- Step 2..N TTFT 降低可在 synthetic benchmark 观测。

### C2. Relay/非连续复用降级为“默认关闭的 experimental”

任务：

- 当前 `RelayRecompute` 不具备 RFC 的 correctness 证明，默认关闭。
- 给 `ReusePipeline` 增加 `ReuseDecision`：`EXACT|PROVABLE_VROPE|APPROX_VERIFY|FULL_RECOMPUTE`。
- 非 exact 情况先统一走 `FULL_RECOMPUTE`，保证 correctness baseline。

验收：

- 所有不满足 exact prefix 的情况不会“裸拼接”旧 KV。

### C3. Virtual-RoPE / KV Validity Check 设计落地

任务：

- 在 `BlockRef.virtual_offset` 与 state 版本链基础上实现 VPV 数据结构。
- 先实现 CPU/PyTorch reference attention 验证 Virtual-RoPE 可证条件 C1/C2/C3。
- 设计接口：`analyze_divergence(new_tokens, prev_state) -> {provable, vpv, reason}`。
- 实现 `estimate_attention_drift()` 的 conservative 上界；drift 超阈值必须 full recompute。
- CUDA/vLLM kernel 改造作为后续独立任务，先保证 API 与 reference correctness。

验收：

- 纯追加 + 后缀不变 + mask 不变：reference attention 与 full recompute bit-exact 或在 dtype 容差内。
- middle/head insertion 默认 fallback，除非显式 experimental。

## 阶段 E：EPD real worker 正确性与性能（P1/P2，预计 4-7 天）

### E1. Qwen3-VL worker 兼容层

任务：

- 锁定 transformers/Qwen3-VL API 版本，添加 capability probe。
- `EncoderWorker` 输出 DeepStack FeatureBundle 时记录 feature shapes、目标 LLM layer、dtype/device。

### E2. Decode 不再全量 materialize 作为最终路径

任务：

- 本地 HF decode 可保留为 correctness reference。

验收：

- decode append 产生新 BlockRef，父 state 不被改写。
- fork 后两个分支 decode 首 token 触发 CoW，shared prefix 页 refcount 正确。

## 阶段 F：Agent 协同调度、Offload、A2A（P2，预计 5-8 天）

### F1. Scheduler 引入队列模型与反压

任务：

- WorkerLoad 增加 `arrival_rate/service_rate/rho/p50/p95/p99/deadline_miss`。
- `AgentScheduler` 实现：
  - bounded queue。
  - admission control。
  - high/critical rho 阈值。
  - token bucket 反压。
  - overload 降级序：复用降级 → 压缩升档 → Offload → reject。

### F2. Offload 状态机

## 16. 2026-07-08 新增进展：真实 burst / poisson 压测通过，并补齐 baseline correctness 对比链

### 16.1 新修复：producer 侧“全空 group block 列表被误判 ready”的并发 bug

真实 burst 压测首轮失败时，decode 侧真实崩溃栈最终落在：

- `wait_for_layer_load() -> RuntimeError: P num blocks less than D`

prefill 日志同时出现：

- `local blocks(0) < remote blocks(1) in a KV cache group`

根因：

- `record_send_reqs()` 原先用 `if block_ids:` 判定 producer transfer 是否 ready；
- 在真实 chunked/layered 并发路径中会出现 `block_ids=[[], []]` 这种“group 已创建但尚未拿到真实 block”的中间态；
- `[[], []]` 在 Python 中是 truthy，导致 connector 过早 `ready.set()`；
- consumer 发起 layered receive 时，remote 侧已有目标 block，而 producer 本地仍是空 group，最终触发 `P num blocks less than D` 并打死 decode engine。

修复：

- 改为只有 `block_ids` 中**至少一个 group 含真实 block**时才 `ready.set()`；
- 对 `[[]]` / `[[], []]` 只保留 pending placeholder，不提前宣布 ready。

新增回归：

### 21.2 实现语义

目标是在多模态 Omni 模型中减少重复 Encoder 计算，尤其是 Agent 多轮/多分支场景里的多模态前缀复用。
