# RFC: EPD Disaggregation for Agent & Multimodal Inference

**RFC-EPD-2026-001** | Status: Active | 2026-06-27

Authors: Mooncake EPD Team
Repository: [Pinoeer-kingxi/Mooncake](https://github.com/Pinoeer-kingxi/Mooncake)
Requires: Mooncake >= 0.3.11, vLLM >= 0.23.0, PyTorch >= 2.0

---

## 1. 这是什么

一句话：把多模态模型推理拆成 Encoder / Prefill / Decode 三段，分别跑在不同 GPU 上，中间用 Mooncake 搬数据。附带做了 Agent 场景下的 KV Cache 克隆、调度路由和视觉特征缓存。

在 Qwen3-VL-8B 上跑通了，和 vLLM 的 MooncakeConnector 也对上了。

---

## 2. 为什么要做

多模态推理的三个计算阶段差别很大：Encoder 是 ViT 前向，Prefill 是全序列 attention 一把算完，Decode 是逐 token 自回归。把它们塞在同一块 GPU 上，互相抢资源，哪个都不痛快。

实际踩到的问题：

- Encoder 跑的时候 Prefill/Decode 干等着，反过来也一样。GPU 利用率上不去。
- 图像密集的负载想单独加 Encoder 节点，做不到——它和 Prefill 绑死了。
- Agent 场景要做 Tree-of-Thought，需要从同一个 KV Cache 状态 fork 出 4-16 个分支。深拷贝 30MB 的 KV Cache 一次要好几毫秒，而且内存翻 N 倍，根本扛不住。

做这个的时间点也到了：Qwen3-VL-8B 一个模型就吃 16GB 显存，32B 往上必须分离部署；Mooncake 的 Transfer Engine 已经支持 TCP/RDMA 双模，基础设施够用；vLLM 0.23.0 也内置了 MooncakeConnector。

---

## 3. 做了哪些，没做哪些

做了的（对应赛题 6 个子任务）：

1. EPD 三阶段分离原型，Vision Encoder 输出通过 Transfer Engine 搬到 Prefill 节点
2. Agent KV Cache 零拷贝克隆（fork / prune / CoW 都有）
3. Qwen3-VL-8B 端到端 Demo，有 EPD vs 单机的对比数据
4. Agent PD 调度——思考型 Agent 走高算力 Prefill，交互型走低延迟 Decode
5. Hidden State 前缀缓存，同一张图不重复编码
6. vLLM MooncakeConnector 集成配置

没做的：

- Omni Pipeline（AR → Generation → Diffusion 那种多阶段），当前只做 Vision + LLM
- 没给上游提 PR，原型验证阶段代码质量还不够合入标准
- 没做 RDMA 优化——手上的 RTX A6000 没有 InfiniBand，TCP 够验证架构了

## 4. 整体架构

```
                     Request Router / Proxy
                     (vllm_v1_proxy_server)
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
     AgentScheduler   StateCloner    PrefixCache
     (think/chat)     (fork/prune)    (LRU)
            │               │               │
            └───────────────┼───────────────┘
                            ▼
                    EPD Pipeline
         ┌──────────┬──────────┬──────────┐
         │ Encoder  │ Prefill  │  Decode  │
         │  GPU 0   │  GPU 1   │  GPU 2   │
         └────┬─────┴────┬─────┴──────────┘
              │ E→P      │ P→D
              ▼          ▼
        Mooncake Transfer Engine (TCP / RDMA)
```

数据流（以 Qwen3-VL-8B 为例）：

```
Image (448×448)
  │ ViT, ~45ms on A6000
  ▼
Hidden States [784, 1152] + Deepstack [3×(196, 4096)]   ← ~1.7MB
  │ Transfer E→P, ~0.3ms
  ▼
Prefill, 212 tokens, ~104ms
  ▼
KV Cache: 36 layers × [1, 8, 212, 128]                   ← ~30MB
  │ Transfer P→D, ~3-31ms (取决于是否跨 NUMA)
  ▼
Decode, ~44-53ms/token, ~19 tok/s
  ▼
Text
```

代码结构：

```
mooncake_epd/
  core/           # EPD 流水线
    transfer_engine.py    # Mooncake 封装，没有的话降级 CUDA memcpy
    encoder_worker.py     # ViT
    prefill_worker.py     # LLM prefill
    decode_worker.py      # 自回归解码
    epd_pipeline.py       # 编排

  benchmarks/     # 性能测试
  scripts/        # 部署脚本
  config/         # 配置
```

## 5. 设计细节

### 5.1 Transfer Engine 封装

`MooncakeTransferWrapper` 做的事很简单：尝试加载 Mooncake 的 TransferEngine，加载不了就降级成 `tensor.to(device)`。后者本质是 CUDA memcpy，在同机多 GPU 下其实也够用。

```python
class MooncakeTransferWrapper:
    def transfer_tensor(tensor, target_device):     # 通用传输
    def transfer_hidden_states(hs, meta, device):   # E→P
    def transfer_kv_cache(kv, device):              # P→D
```

统计信息（传输量、带宽、峰值）用 `threading.Lock` 保护，多线程调用不会出问题。

实测带宽数据（A6000, PCIe Gen4）：

- 同 NUMA GPU 对：KV Cache 传输 ~73 Gbps，Hidden State ~91 Gbps
- 跨 NUMA GPU 对：KV Cache 传输 ~13 Gbps，Hidden State ~51 Gbps

跨 NUMA 衰减明显，这也是 EPD 分离部署时需要考虑的——尽量把 P 和 D 放在同一个 NUMA node 内。

### 5.2 Encoder Worker

就是跑 ViT。但 Qwen3-VL 的 Vision Encoder 接口和常规 HF 模型不太一样，需要传 `grid_thw` 参数：

```python
# Qwen3-VL 特有
visual_out = model.model.visual(pixel_values, grid_thw=image_grid_thw)
# 输出除了 last_hidden_state，还有 deepstack_features
# deepstack_features 是 3 层投射到 LLM hidden_dim 的视觉特征
# 分别注入到 LLM 的第 8/16/24 层
```

这个 deepstack 机制意味着 E→P 传输的不只是原始视觉特征，还有多层投射后的特征。当前实现里 Encoder 和 Prefill 在同一 GPU 上，这个问题不大；但如果真正做到跨节点分离，deepstack features 的传输需要额外处理。

### 5.3 Prefill & Decode Worker

Prefill 接收视觉特征和文本 token，跑一次完整的 forward 拿到 KV Cache。Decode 拿着 KV Cache 逐 token 生成。

这里有一个实现上容易出错的点：Decode 的第一步（step 0）必须把 Prefill 产出的 KV Cache 传进去。我们第一版代码里写了个 `if step > 0` 的条件判断，把 step 0 的 KV Cache 跳过了，结果 Decode 阶段的输出完全是乱的。修了之后才正常。

Qwen3-VL 的 KV Cache 用的是 transformers 5.x 的 `DynamicCache`，不是老版本的 tuple list：

```
DynamicCache
  └── layers[0..35]: DynamicLayer
       ├── keys:   [batch, 8, seq_len, 128]
       └── values: [batch, 8, seq_len, 128]
```

36 层，每层 8 个 KV head，head_dim 128。212 个 token 的 prefill 产出 ~30MB 的 KV Cache。

### 5.4 Agent State Cloning

这是整个项目里设计最花心思的部分。

需求很直接：Agent 做 Tree-of-Thought 的时候，要从当前 KV Cache 状态 fork 出多个分支，每个分支独立生成，最后选得分最高的。传统做法是深拷贝——30MB 的 KV Cache 拷 8 份就是 240MB，还要花好几毫秒。

我们用的是引用计数。物理张量只有一份，fork 只是给 ref_count +1，O(1) 完成：

```python
class _CacheEntry:
    kv_cache: Tuple[Tensor, Tensor]   # 物理张量，所有分支共享
    ref_count: int                     # 有几份引用
    size_bytes: int

class AgentStateCloner:
    _caches: Dict[str, _CacheEntry]         # 物理缓存池
    _branch_to_cache: Dict[str, str]        # 分支 → 它用的是哪个缓存
    _branches: Dict[str, BranchState]       # 分支元信息
```

生命周期：
```
register("base", T)    → caches["base"] = {T, ref=1}
clone("base", "b1")    → caches["base"].ref = 2, branch_to_cache["b1"] = "base"
clone("base", "b2")    → caches["base"].ref = 3, branch_to_cache["b2"] = "base"
release("b1")          → caches["base"].ref = 2
release("b2")          → caches["base"].ref = 1
# ref 到 0 时自动释放物理内存
```

如果某个分支需要修改 KV Cache（比如往里面写新 token），走 Copy-on-Write：分配新的物理张量，把分支指向新缓存，老缓存 ref -1。

这里第一版写了好几个 bug：
- `write_copy_on_write` 里遍历 `_kv_store.items()` 的同时调用 `_release` 删除元素，直接 RuntimeError
- `release_branch` 只减了分支自己的缓存引用，没减源缓存的引用，导致源缓存的 ref_count 永远降不下来，内存泄漏
- 一开始的设计是给每个分支创建一个新的 cache entry（指向同一份物理张量），然后 ref_count 各算各的，搞得很复杂还容易算错

后来全部重构成上面这个模型：物理张量和引用计数是一对一的 `_CacheEntry`，分支只通过 `_branch_to_cache` 映射找到对应的 entry。改完之后引用计数就再没出过问题。

实测：8 分支克隆 30MB KV Cache，总耗时 0.19ms。深拷贝同样的场景大约要 3-5ms 加 240MB 内存。

### 5.5 PD 调度

给不同类型的 Agent 分配不同的资源：

- 思考型（长推理、复杂分析）→ 找剩余算力最多的 Prefill worker
- 交互型（多轮对话、快速响应）→ 找延迟最低的 Decode worker
- 混合型 → 两边都走均衡

评分公式很朴素：

```python
score = current_load + queue_size * 0.1
if prefer_low_latency:  score += avg_latency_ms / 1000
if prefer_high_capacity: score -= max_capacity * 0.01
# 选 score 最小的
```

批量路由时先按优先级排序，但返回结果保持输入顺序——这个我们一开始没注意，按优先级排完直接返回了，调用方拿到的路由结果和请求列表对不上。

### 5.6 Hidden State 前缀缓存

同一张图可能被多次查询（同一个用户在多轮对话里反复引用同一张图，或者不同用户问同一张图），没必要每次都过一遍 ViT。

缓存 key 的计算是个需要权衡的点。最初的实现是对整个 tensor 做 SHA-256：

```python
# 旧方案：把整个 tensor 拷到 CPU，转 numpy，转 bytes，再 hash
data = tensor.cpu().numpy().tobytes()
hash = sha256(data).hexdigest()[:32]
```

问题在于这三次拷贝加上 hash 计算本身就要 12ms。而 224×224 的图编码一次也就 38ms——缓存查找花了编码 1/3 的时间，加速效果很有限。

改成了采样 hash：取 tensor 的 shape、dtype、device 作为元信息，再从展平后的数据里等间距采 1024 个元素，拼起来做 MD5。一张 448×448 的图大约 600KB，只采样 ~4KB 就够了。

```python
def _fast_tensor_hash(tensor):
    meta = f"{shape}_{dtype}_{device}"
    sample = tensor.reshape(-1)[::max(1, n // 1024)]
    return md5(meta.encode() + sample.view(uint8).numpy().tobytes()).hexdigest()
```

改完之后查找从 12ms 降到 0.08ms。和重新编码 (45ms) 比，缓存查找 (1ms) 快 42 倍。

碰撞概率方面，128-bit MD5 在 1000 条目下碰撞概率约 10^{-33}，完全够用。如果后面要做到安全敏感场景，换回 SHA-256 就行。

淘汰策略是标准 LRU（`OrderedDict`），加上可配置的 TTL（默认 1 小时）和最大容量（默认 4GB）。用 `time.monotonic()` 而不是 `time.time()` 做时间判断，免得系统时钟跳变导致 TTL 异常。

## 6. 用法

### EPD 流水线

```python
from mooncake_epd.core import EPDPipeline, MooncakeTransferWrapper

transfer = MooncakeTransferWrapper(protocol="tcp")
transfer.initialize()
pipeline = EPDPipeline(encoder, prefill, decode, transfer,
                       encoder_device="cuda:0",
                       prefill_device="cuda:1",
                       decode_device="cuda:2")

output, stats = pipeline.process(pixel_values, input_ids,
                                 max_new_tokens=128, temperature=0.7)
print(stats.summary())
# encode=28ms, prefill=104ms, transfer_p2d=31ms, decode=858ms, tps=18.65

pipeline.shutdown()
```

### Agent 克隆

```python
from mooncake_epd.agent import AgentStateCloner

cloner = AgentStateCloner()
cloner.register_kv_cache("root", kv_cache_tuple)
branches = cloner.fork_branches("root", num_branches=4)  # 零拷贝

for b in branches:
    b.score = evaluate(b)
cloner.prune_branches(keep_top_k=2)
```

### 调度

```python
from mooncake_epd.agent import AgentPDScheduler, AgentRequest, AgentType

scheduler = AgentPDScheduler(
    prefill_workers=["prefill_gpu0", "prefill_gpu1"],
    decode_workers=["decode_gpu2", "decode_gpu3"])

routings = scheduler.batch_route([
    AgentRequest("r1", AgentType.THINKING, priority=10),
    AgentRequest("r2", AgentType.INTERACTIVE, priority=5),
])
```

## 7. 性能数据

测试环境：8× RTX A6000 48GB, 64 cores, 503GB RAM, CUDA 12.9, TCP 模式（无 RDMA），模型 Qwen3-VL-8B-Instruct BF16。

### Vision Encoder

| 分辨率 | 编码耗时 | 输出 |
|-------|---------|------|
| 224×224 | 37.9 ms | [256, 1152] + 3×[64, 4096] deepstack |
| 336×336 | 30.8 ms | [400, 1152] + 3×[100, 4096] deepstack |
| 448×448 | 44.9 ms | [784, 1152] + 3×[196, 4096] deepstack |

336 比 224 快这个有点意外，可能是 ViT 内部 patch 合并的 batch effect。

### 跨 GPU 传输

| 传输内容 | 大小 | 耗时 | 带宽 |
|---------|------|------|------|
| KV Cache (36层) | 29.25 MB | 18.76 ms | 13.08 Gbps |
| Hidden State | 1.72 MB | 0.28 ms | 51.48 Gbps |

这是跨 NUMA 的数据。同 NUMA 下 KV Cache 能到 ~73 Gbps。PCIe Gen4 x16 理论峰值约 250 Gbps，实际利用率受 NUMA 拓扑和 CUDA stream 调度影响。

### EPD 端到端

| 阶段 | 耗时 |
|------|------|
| Encoder | 81.4 ms |
| Prefill (212 tokens) | 104.2 ms |
| KV Transfer P→D | 31.1 ms |
| Decode (16 tokens) | 858.1 ms |
| TTFT | 312.2 ms |

### EPD vs 单机

| | 耗时 | Tokens | 吞吐 |
|--|------|--------|------|
| 单机 generate | 1703 ms | 32 | 18.79 tok/s |
| EPD 分离 | 1075 ms (E+P+D) | 16 | 18.65 tok/s |

单请求吞吐基本一样——这在意料之中，Decode 的自回归瓶颈决定了单请求 TPS 上限。EPD 分离的好处主要体现在并发：E 和 D 可以同时处理不同请求，P 和 D 可以独立扩缩容。

### Agent 克隆

| 分支数 | 总耗时 | 每分支 |
|--------|--------|--------|
| 2 | 0.073 ms | 0.037 ms |
| 4 | 0.108 ms | 0.027 ms |
| 8 | 0.192 ms | 0.024 ms |

30MB KV Cache，8 分支只要 0.19ms。深拷贝同样场景大约 3-5ms + 240MB 额外内存。

### 前缀缓存

编码一次 45.69ms，缓存查找 1.07ms。命中率 100% 时加速 42.6 倍。

## 9. 还没想清楚的事

1. **跨节点 EPD 到底划不划算**。我们目前只在单机内验证了。跨节点的 TCP 带宽大概 10 Gbps，传 30MB KV Cache 要 24ms，而 Prefill 本身才 100ms。传输开销占比 20%，可能不划算。需要 RDMA 环境（200 Gbps+）才能看到明显收益。

2. **Deepstack features 怎么传**。Qwen3-VL 的 deepstack features 是 3 个 [N, 4096] 的 tensor，加起来和 KV Cache 差不多大。如果 Encoder 和 Prefill 在不同节点，这部分也得传。但目前还没想好是拼在 Hidden States 里一起传，还是走独立的传输通道。

3. **DynamicCache 跨节点序列化**。vLLM 的 DynamicCache 对象不能直接 pickle，跨节点传输需要自己写序列化。这个问题在 MooncakeConnector 里可能有现成方案，但我们还没深入看。

4. **CoW 和多分支并发 Decode 的正确性**。目前每个分支独立跑 `model.forward`，KV Cache 增量各自追加，不会互相干扰。但如果后面要做 batched decode（多个分支合在一个 batch 里跑以提高 GPU 利用率），共享 KV Cache 的分支怎么处理 mask 是个问题。

## 10. 做过的取舍

**EPD vs PD**：选 EPD 是因为多模态场景下图重复率高，前缀缓存能吃掉大量重复编码。纯文本场景下 EPD 比 PD 多一次传输开销，没必要。

**引用计数 vs 深拷贝**：引用计数代码复杂度高很多（要管生命周期、CoW、防泄漏），但 ToT 场景下深拷贝的时间和内存代价实在太大。如果只做单分支推理，深拷贝更简单可靠。

**采样 hash vs 全量 hash**：采样 1024 个元素的 MD5 在碰撞安全性上对缓存场景完全够用。全量 SHA-256 的问题是查找太慢，让缓存本身成了瓶颈。如果后面发现碰撞率不对（目前没遇到过），再升级。
