## 1. Summary

本 RFC 提出一种面向多模态大语言模型（MLLM）推理的 **EPD 三阶段分离架构**（Encoder-Prefill-Decode），将传统 PD（Prefill-Decode）分离进一步扩展至 Encoder 阶段，并利用 Mooncake Transfer Engine 实现跨 GPU / 跨节点的高性能数据传输。

同时，本框架引入了 **Agent 状态协同** 能力，包括：
- **零拷贝 KVCache 克隆**：支持 Tree-of-Thought 等并行推理场景
- **Agent PD 调度策略**：根据任务类型（思考型 / 交互型 / 混合型）动态路由
- **Hidden State 前缀缓存**：避免重复 Vision Encoder 计算

本框架已在 **Qwen3-VL-8B-Instruct** 真实模型上完成端到端验证，并与 vLLM v0.23.0 的 `MooncakeConnector` 完成集成。

## 2. Motivation

### 2.1 问题陈述

多模态大模型推理包含三个计算特性迥异的阶段：

| 阶段 | 计算特征 | 资源需求 | 延迟特征 |
|------|----------|----------|----------|
| **Encoder (E)** | CNN/ViT 前向，batch 并行 | GPU 算力密集，显存中等 | 固定延迟 (20-50ms) |
| **Prefill (P)** | 全序列 Attention，一次计算 | GPU 算力+显存密集 | 与序列长度线性 |
| **Decode (D)** | 逐 token 自回归 | GPU 算力低，显存带宽密集 | 延迟敏感 |

传统单机部署将三个阶段耦合在同一 GPU 上，导致：

1. **资源利用率低**：Encoder 计算时 Prefill/Decode 资源闲置，反之亦然
2. **无法独立扩缩容**：高图像输入场景无法单独扩容 Encoder 节点
3. **Agent 场景瓶颈**：多分支推理（Tree-of-Thought）需要克隆 KV Cache，单机无法高效支持跨节点状态共享

### 2.2 为什么现在？

- **模型规模增长**：Qwen3-VL-8B (16GB FP16) 已接近单卡极限，32B+ 模型必须分离部署
- **Agent 工作流兴起**：多轮推理、A2A 状态共享成为新范式
- **Mooncake 成熟**：Transfer Engine 支持 TCP/RDMA 双模传输，KVCache 池化能力完备
- **vLLM 原生支持**：v0.23.0 已注册 `MooncakeConnector`，提供 KV transfer 基础设施

## 4. Background

### 4.1 PD Disaggregation (Prior Art)

传统 PD 分离将 LLM 推理分为：
- **Prefill 节点**：处理 prompt，生成完整 KV Cache
- **Decode 节点**：接收 KV Cache，逐 token 自回归生成

代表工作：Mooncake + vLLM `MooncakeConnector`、Splitwise、DistServe。

### 4.2 EPD Extension (This Work)

我们将分离扩展至三阶段：

```
传统 PD:   [Vision Encoder + Prefill] → KV Cache → [Decode]
本方案 EPD: [Vision Encoder] → Hidden States → [Prefill] → KV Cache → [Decode]
```

**关键差异**：
- Encoder 独立部署，可针对图像密集型负载独立扩缩
- E→P 传输 Hidden States（通常 1-5MB），P→D 传输 KV Cache（通常 10-100MB）
- 两段传输的数据格式和粒度不同，需要差异化的传输策略

### 4.3 Agent State Management

Tree-of-Thought、Beam Search 等推理策略需要从同一 KV Cache 状态 fork 出多个并行分支。传统做法是深拷贝 KV Cache（O(n) 时间和空间），本方案通过引用计数实现零拷贝克隆（O(1)）。

## 6. Detailed Design

### 6.1 Transfer Engine (`core/transfer_engine.py`)

**设计原则**：封装 Mooncake Transfer Engine，提供类型安全的 GPU Tensor 传输接口，在不具备 RDMA 硬件的环境下优雅降级至 TCP/CUDA memcpy。

```python
class MooncakeTransferWrapper:
    """
    核心传输接口。

    - initialize(): 尝试加载 Mooncake Transfer Engine，失败时降级为本地模式
    - transfer_tensor(): 通用 Tensor 传输（保留 dtype，支持 bf16/fp16/fp32）
    - transfer_hidden_states(): E→P 阶段专用
    - transfer_kv_cache(): P→D 阶段专用
    """
```

**线程安全**：`TransferStats` 使用 `threading.Lock` 保护计数器，支持多线程并发传输。

**降级策略**：
1. 尝试 `from mooncake.transfer_engine import TransferEngine`
2. `ImportError` → 降级为 `tensor.to(target_device, copy=True)` (CUDA memcpy)
3. 记录 warning 日志，不影响流水线运行

### 6.2 Encoder Worker (`core/encoder_worker.py`)

**职责**：在独立 GPU 上运行 Vision Encoder，输出 Hidden States。

```python
class EncoderWorker:
    @torch.no_grad()
    def encode_images(pixel_values, image_sizes) -> EncoderOutput:
        """
        支持多种模型接口：
        - model.visual(pixel_values)              # 标准 ViT
        - model.visual(pixel_values, grid_thw=gt) # Qwen3-VL
        - model.vision_model(pixel_values)         # HF 通用
        """
```

**Qwen3-VL 适配**：
- 输入：`pixel_values [N, 1536]` (展平的 patch tokens) + `image_grid_thw [B, 3]`
- 输出：`last_hidden_state [N, 1152]` + `deepstack_features [3 × (M, 4096)]`
- Deepstack features 是投射到 LLM hidden_dim 的多层视觉特征

### 6.3 Prefill Worker (`core/prefill_worker.py`)

**职责**：接收视觉特征和文本 token，执行完整 Prefill 计算，生成 KV Cache。

**关键设计**：
- 视觉特征通过 `vision_proj` (Linear) 投射到 LLM hidden_dim 后拼接到文本 embedding 前面
- `use_cache=True` 使模型输出 `past_key_values`
- KV Cache 从 `DynamicCache.layers[i].keys/.values` 提取并 stack

**Qwen3-VL KV Cache 格式**：
```
DynamicCache (36 layers)
  └── layers[i]: DynamicLayer
       ├── keys:  [batch, 8, seq_len, 128]   # 8 KV heads, 128 head_dim
       └── values: [batch, 8, seq_len, 128]
```

### 6.4 Decode Worker (`core/decode_worker.py`)

**职责**：接收 KV Cache，逐 token 自回归解码。

**关键修复**：
- **Bug 修复**：step 0 时正确传入 Prefill KV Cache（此前被跳过）
- **采样一致性**：首 token 和后续 token 使用相同的 temperature 参数

```python
class DecodeWorker:
    @torch.no_grad()
    def decode(input_ids, kv_cache, max_new_tokens, temperature) -> DecodeOutput:
        past_kv = _rebuild_past_key_values(kv_cache)  # stack → list
        for step in range(max_new_tokens):
            outputs = model(input_ids, past_key_values=past_kv, use_cache=True)
            past_kv = outputs.past_key_values  # 增量更新
            if temperature > 0:
                next_token = multinomial(softmax(logits / temperature))
            else:
                next_token = argmax(logits)
```

### 6.5 Agent State Cloning (`agent/state_clone.py`)

**设计原则**：使用引用计数实现零拷贝 KV Cache 克隆，多个分支共享同一份物理内存。

**核心数据结构**：

```python
@dataclass
class _CacheEntry:
    kv_cache: Tuple[Tensor, Tensor]  # 物理张量（共享）
    ref_count: int                    # 引用计数
    size_bytes: int                   # 物理内存大小

class AgentStateCloner:
    _caches: Dict[str, _CacheEntry]        # cache_id → 物理缓存
    _branch_to_cache: Dict[str, str]       # branch_id → cache_id
    _branches: Dict[str, BranchState]      # branch_id → 分支状态
```

**引用计数模型**：
```
register("base", T)           → _caches["base"] = {T, ref=1}
clone("base", "b1")           → _caches["base"].ref = 2, _branch_to_cache["b1"] = "base"
clone("base", "b2")           → _caches["base"].ref = 3, _branch_to_cache["b2"] = "base"
release("b1")                 → _caches["base"].ref = 2
release("b2")                 → _caches["base"].ref = 1
release base (implicit)       → _caches["base"].ref = 0 → 自动释放
```

**写时复制 (CoW)**：
```python
def write_copy_on_write(branch_id, new_kv_cache):
    old_cache_id = _branch_to_cache[branch_id]
    _caches[new_id] = _CacheEntry(new_kv_cache, ref=1)
    _branch_to_cache[branch_id] = new_id
    _decrement_ref(old_cache_id)  # 减少旧缓存引用
```

**已修复的关键 Bug**：
1. ~~`write_copy_on_write` 遍历字典时删除元素 → RuntimeError~~ → 使用 `_branch_to_cache` 映射
2. ~~`release_branch` 不减少 source 引用计数 → 内存泄漏~~ → 精确追踪 branch→cache 映射
3. ~~引用计数双重计算~~ → 统一为单一 `_CacheEntry` 模型

### 6.6 Agent PD Scheduler (`agent/scheduler.py`)

**路由策略**：

| Agent 类型 | Prefill 选择 | Decode 选择 | 场景 |
|-----------|-------------|------------|------|
| THINKING | 最高算力 (prefer_high_capacity) | 默认 | 长推理、复杂分析 |
| INTERACTIVE | 最低延迟 (prefer_low_latency) | 最低延迟 | 多轮对话、快速响应 |
| HYBRID | 均衡 | 低延迟 | 混合负载 |

**评分函数**：
```python
score = current_load + queue_size × 0.1
if prefer_low_latency:
    score += avg_latency_ms / 1000
if prefer_high_capacity:
    score -= max_capacity × 0.01
```

**batch_route 保序**：按优先级排序后路由，返回结果按原始输入顺序排列。

### 6.7 Hidden State Prefix Cache (`agent/prefix_cache.py`)

**设计目标**：缓存 Vision Encoder 输出，避免相同图像的重复编码。

**快速 Hash 算法**：
```python
def _fast_tensor_hash(tensor):
    """采样 1024 个元素 + 元信息 → MD5，避免全量数据拷贝"""
    meta = f"{shape}_{dtype}_{device}"
    flat = tensor.reshape(-1)
    sample = flat[::max(1, n // 1024)]
    data = meta.encode() + sample.view(uint8).numpy().tobytes()
    return md5(data).hexdigest()
```

**性能对比**：

| Hash 方法 | 448×448 图像 | 加速比 |
|-----------|-------------|--------|
| SHA-256 全量 (旧) | 12.3ms | 基准 |
| MD5 采样 (新) | 0.08ms | **154×** |
| 缓存查找 vs 重新编码 | 1.07ms vs 45.7ms | **42.6×** |

**LRU 淘汰策略**：
- `OrderedDict` 实现 O(1) 访问 + O(1) LRU 淘汰
- 可配置：最大缓存大小 (默认 4GB)、最大条目数 (默认 1000)、TTL (默认 1h)
- 使用 `time.monotonic()` 避免系统时钟跳变影响

cloner = AgentStateCloner()

# Fork 4 thinking branches (zero-copy)

# Assign scores from parallel decoding

    branch.score = evaluate(branch)

### 8.1 Test Environment

| Component | Specification |
|-----------|--------------|
| GPU | 8× RTX A6000 (48GB GDDR6) |
| CPU | 64 cores |
| RAM | 503 GB |
| CUDA | 12.9 |
| Protocol | TCP (no RDMA) |
| Model | Qwen3-VL-8B-Instruct (BF16) |

### 8.2 Vision Encoder Performance

| Resolution | Encode Time | Hidden Shape | Deepstack |
|-----------|------------|-------------|-----------|
| 224×224 | **37.9 ms** | [256, 1152] | 3×[64, 4096] |
| 336×336 | **30.8 ms** | [400, 1152] | 3×[100, 4096] |
| 448×448 | **44.9 ms** | [784, 1152] | 3×[196, 4096] |

### 8.3 Cross-GPU Transfer Bandwidth

| Data Type | Size | Transfer Time | Bandwidth |
|-----------|------|--------------|-----------|
| KV Cache (36 layers) | 29.25 MB | 18.76 ms | **13.08 Gbps** |
| Hidden State | 1.72 MB | 0.28 ms | **51.48 Gbps** |

> Note: PCIe Gen4 x16 theoretical peak ~250 Gbps. Achieved 5-20% utilization on cross-NUMA GPU pairs. Same-NUMA pairs achieve 73-91 Gbps.

### 8.4 EPD Pipeline End-to-End

| Stage | Time | Notes |
|-------|------|-------|
| Encoder (E) | 81.4 ms | ViT 448×448 |
| Prefill (P) | 104.2 ms | 212 input tokens |
| Transfer P→D | 31.1 ms | 29.8 MB KV Cache |
| Decode (D) | 858.1 ms | 16 tokens, 18.65 tok/s |
| **TTFT** | **312.2 ms** | E + P + Transfer + 1st token |

### 8.5 Single Node vs EPD Comparison

| Mode | Total Time | Tokens | Throughput |
|------|-----------|--------|-----------|
| Single Node (generate) | 1703 ms | 32 | **18.79 tok/s** |
| EPD Disaggregated | 1075 ms (E+P+D) | 16 | **18.65 tok/s** |

**分析**：
- 单请求吞吐量基本一致 (18.79 vs 18.65 tok/s)，符合预期——Decode 阶段的自回归特性决定了单请求 TPS 受限于模型计算
- EPD 分离的价值在于 **并发场景**：
  - E 和 D 可并行处理不同请求
  - P 和 D 可独立扩缩容
  - 高并发时流水线填充率更高

## 10. Security Considerations

| Threat | Mitigation |
|--------|-----------|
| 未授权访问 Mooncake Master | 生产环境应启用 TLS，限制 RPC 端口绑定 |
| KV Cache 数据泄露 | Transfer Engine 支持加密传输；缓存数据不持久化到磁盘 |
| 缓存投毒 (Prefix Cache) | Hash 碰撞概率极低 (MD5 128-bit)；生产环境可升级为 SHA-256 |
| 调度器 DoS | `MAX_ROUTING_LOG` 限制日志增长；Worker 容量检查 |
| GPU 内存耗尽 | `gpu_memory_utilization` 限制；LRU 淘汰防止缓存溢出 |

## 11. Open Questions and Future Work

### 11.1 Open Questions

1. **跨节点 EPD**：当前验证仅限单机多 GPU。跨节点场景下 TCP 带宽 (~10 Gbps) 是否成为瓶颈？需要 RDMA 环境验证。

2. **Deepstack Features 传输**：Qwen3-VL 的 `deepstack_features` (3×[N, 4096]) 是否也需要通过 Transfer Engine 传输？当前实现中 Prefill 在同一 GPU 上可直接访问，分离部署时需要额外传输。

3. **DynamicCache 跨节点序列化**：vLLM 的 `DynamicCache` 对象无法直接序列化。跨节点传输需要自定义序列化协议。

4. **CoW 在 Decode 阶段的正确性**：当多个分支共享 KV Cache 并同时 Decode 时，需要确保每个分支的 KV Cache append 操作不互相干扰。当前实现中每个分支独立调用 `model.forward`，KV Cache 增量追加在各自的 `DynamicCache` 实例上。

## 12. Alternatives Considered

### 12.1 EPD vs PD

| 方案 | 优势 | 劣势 |
|------|------|------|
| **EPD (本方案)** | Encoder 独立扩缩容；视觉特征可缓存 | 增加 E→P 传输开销；需要处理 deepstack features |
| PD (传统) | 架构简单；无 E→P 传输 | Encoder 和 Prefill 耦合，无法独立优化 |

**选择理由**：多模态场景下图像输入量大且重复率高（同一图像多次查询），EPD 分离配合前缀缓存可显著降低 Encoder 计算开销（42.6× 加速）。

### 12.2 引用计数 vs 深拷贝

| 方案 | Clone 时间 | 内存 | 复杂度 |
|------|-----------|------|--------|
| **引用计数 (本方案)** | O(1) ~0.02ms | 共享 | 需要 CoW、生命周期管理 |
| 深拷贝 | O(n) ~ms 级 | N 倍 | 简单，无共享风险 |

**选择理由**：Tree-of-Thought 场景通常需要 4-16 个分支，深拷贝 30MB KV Cache × 16 = 480MB 内存 + 数毫秒延迟，而引用计数仅需 <0.2ms 且内存不增长。

### 12.3 采样 Hash vs 全量 Hash

| 方案 | 查找时间 | 碰撞风险 | 适用场景 |
|------|---------|---------|---------|
| **MD5 采样 (本方案)** | 0.08ms | 极低 (10⁻³³) | 高频查询、大图 |
| SHA-256 全量 | 12.3ms | 极低 | 安全敏感场景 |

**选择理由**：前缀缓存的核心价值是加速。全量 hash 12.3ms 已超过部分分辨率的编码时间 (30ms)，使缓存几乎无收益。采样 hash 在保持碰撞安全性的同时将查找降至 0.08ms，实现 42.6× 加速。
