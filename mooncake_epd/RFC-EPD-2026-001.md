## 1. 问题定义

多模态大语言模型（MLLM）的推理过程天然包含三个计算特征差异显著的阶段：Vision Encoder 将图像编码为视觉特征（Encoder），大语言模型对完整输入序列执行一次前向计算生成 KV Cache（Prefill），随后逐 token 自回归生成文本（Decode）。传统 PD（Prefill-Decode）分离架构将 Encoder 与 Prefill 耦合在同一计算节点，忽略了视觉编码阶段在计算模式、资源需求和可缓存性方面的独特属性。

与此同时，Agent 工作流对推理基础设施提出了新的要求。Tree-of-Thought、Beam Search 等推理策略需要从同一中间状态派生出多个并行分支，而多轮对话、工具调用等场景则要求系统能够根据任务特征将请求调度到不同类型的计算资源上。

本方案围绕赛题的六项任务，提出一套以 Mooncake Transfer Engine 为传输底座的 EPD 三阶段分离框架，并在此基础上构建 Agent 状态协同能力。

## 2. 设计思路

### 2.1 从 PD 分离到 EPD 分离

PD 分离的核心动机是 Prefill 和 Decode 两个阶段的计算特性不匹配：Prefill 是计算密集型（大矩阵乘法，高 GPU 利用率），Decode 是访存密集型（逐 token 生成，受限于显存带宽）。将两者分离到不同节点，可以分别针对各自特性优化硬件配置和调度策略。

EPD 分离在此基础上进一步拆分出 Encoder 阶段。这一决策的依据在于视觉编码阶段具有三个区别于 Prefill 的特征：

第一，计算模式不同。Vision Encoder（通常为 ViT 架构）执行的是标准的前馈网络计算，不涉及 KV Cache 的生成与管理，其 GPU 利用模式与 LLM 的 attention 计算存在显著差异。

第二，延迟可预测。给定输入图像分辨率，Vision Encoder 的计算量是确定的，不像 Prefill 那样随输入序列长度线性增长。这使得 Encoder 节点可以更精确地进行容量规划。

第三，输出可缓存。同一张图像在不同请求中可能反复出现（例如多轮对话中用户反复引用同一张图片），其编码结果可以复用而无需重新计算。这一点在 PD 架构中无法利用，因为 Encoder 输出与 Prefill 计算耦合在同一节点上。

基于上述分析，EPD 分离的数据通路设计为两段独立的传输链路：

- **E→P 链路**：传输 Vision Encoder 输出的 Hidden States。数据量相对较小（通常在 1-5 MB 量级），但对延迟敏感，因为 Prefill 阶段需要等待 Hidden States 到达后才能开始计算。
- **P→D 链路**：传输 Prefill 产出的 KV Cache。数据量较大（通常在 10-100 MB 量级，随模型层数和序列长度增长），需要高带宽传输能力。

两段链路的数据粒度、传输时机和优化策略各不相同，需要在 Transfer Engine 层面提供差异化的传输接口。

### 2.2 Agent 状态协同的设计考量

Agent 工作流对 KV Cache 管理提出了两类需求：

**状态克隆（State Cloning）**。在 Tree-of-Thought 等并行推理场景中，Agent 需要从当前推理状态派生出多个分支，每个分支独立生成后续内容，最终选择最优路径。这要求系统能够在不复制物理内存的前提下创建 KV Cache 的逻辑副本，并在分支不再需要时正确回收资源。

**差异化调度（Disaggregated Scheduling）**。不同类型的 Agent 任务对计算资源的需求不同。需要深度推理的"思考型" Agent 倾向于分配到高算力的 Prefill 资源（因为其 Prompt 通常较长且复杂），而需要快速响应的"交互型" Agent 则更适合分配到低延迟的 Decode 资源。调度器需要根据任务类型动态路由请求。

这两类需求共同指向一个核心设计问题：如何在分布式环境中管理 KV Cache 的生命周期和访问模式。我们的思路是在 Transfer Engine 之上构建一层状态管理层，负责引用计数、写时复制和路由决策，而不直接介入底层的数据传输。

## 3. 架构方案

### 3.1 总体架构

系统分为三层：

- **传输层**：封装 Mooncake Transfer Engine，提供面向 EPD 各阶段数据传输的专用接口。在具备 RDMA 硬件时使用 RDMA 传输，否则降级为 TCP 或 CUDA IPC。
- **流水线层**：编排 Encoder、Prefill、Decode 三个 Worker 的执行顺序和数据流转，管理各阶段的设备分配与同步。
- **协同层**：在流水线之上提供 Agent 状态克隆、PD 调度和前缀缓存能力，面向上层 Agent 框架暴露接口。

三层之间的依赖关系是单向的：协同层依赖流水线层，流水线层依赖传输层。每一层可以独立替换实现——例如传输层可以从 Mooncake 切换为其他传输方案而不影响上层逻辑。

### 3.2 各阶段 Worker 的职责划分

**Encoder Worker** 负责在指定 GPU 上运行 Vision Encoder，将输入的预处理图像张量编码为视觉特征向量。需要注意的是，以 Qwen-VL 为代表的新一代多模态模型，其 Vision Encoder 的输出不仅包含原始的视觉 Hidden States，还包含经过多层投影后注入到 LLM 不同层级的 Deepstack Features。EPD 分离时需要将这些特征一并传输到 Prefill 节点。

**Prefill Worker** 接收来自 Encoder 的视觉特征和文本 token IDs，将视觉特征通过投影层映射到 LLM 的隐藏维度后拼接到文本 Embedding 前部，执行一次完整的 LLM 前向计算（`use_cache=True`），产出 KV Cache 和首 token 的 logits。

**Decode Worker** 接收 Prefill 产出的 KV Cache，以首 token logits 为起点进行自回归解码。每一步生成一个 token，将新的 key/value 追加到 KV Cache 中，直到遇到 EOS token 或达到最大长度。

### 3.3 数据传输策略

EPD 两段传输链路的数据特征差异决定了不同的优化方向：

对于 E→P 传输，Hidden States 的数据量较小但传输频率高（每个多模态请求都需要），优化重点在于降低传输延迟。在同机多 GPU 场景下，可以利用 CUDA IPC 或 NVLink 实现低延迟传输；在跨节点场景下，需要考虑 Hidden States 的压缩传输以减少网络开销。

对于 P→D 传输，KV Cache 的数据量随模型规模线性增长（以 Qwen3-VL-8B 为例，36 层、8 个 KV head、128 维 head_dim，212 个 token 的 KV Cache 约为 30 MB），优化重点在于带宽利用率。Transfer Engine 的 RDMA 模式在此场景下优势最为明显。

### 3.4 与 vLLM 的集成路径

选择 vLLM 作为集成框架，原因在于 vLLM 从 v0.23.0 起已内置 `MooncakeConnector`，提供了 KV Cache 跨节点传输的基础设施，以及 `kv_transfer_config` 配置接口用于指定 Prefill（`kv_producer`）和 Decode（`kv_consumer`）角色。

集成方案是：利用 vLLM 的 MooncakeConnector 处理 P→D 的 KV Cache 传输，同时在 vLLM 的 serving 层之上插入 Encoder Worker 和 E→P 传输逻辑，将 Vision Encoder 的计算从 vLLM 的推理流程中剥离出来。具体而言，Prefill 节点在接收到请求时，先从 Encoder 节点获取 Hidden States，再将其作为 `inputs_embeds` 的一部分传入 LLM 完成 Prefill 计算。

## 4. 关键技术方案

### 4.1 KV Cache 零拷贝克隆

Agent State Cloning 的核心挑战在于：多个推理分支需要共享同一份 KV Cache 物理数据，但每个分支在后续 Decode 过程中会产生独立的 KV Cache 增量。

我们采用引用计数 + 写时复制（Copy-on-Write）的方案：

**引用计数模型**。维护一个物理缓存池，每个物理缓存条目持有一个引用计数。当 Agent fork 出新分支时，新分支指向同一物理缓存，引用计数加一；当分支完成或被剪枝时，引用计数减一；减至零时自动释放物理内存。

这一模型的关键设计决策是：一个物理缓存条目有且只有一个引用计数，所有指向该条目的分支共享这个计数。避免为每个分支创建独立的缓存元数据条目——那样会导致引用计数双重计算和回收逻辑复杂化。

**写时复制**。当某个分支需要修改 KV Cache（即在 Decode 过程中追加新的 key/value），为其分配新的物理缓存，将分支的指向切换到新缓存，同时减少旧缓存的引用计数。这保证了其他共享旧缓存的分支不受影响。

**分支标识与映射**。通过 `branch_id → cache_id` 的映射表追踪每个分支所使用的物理缓存。这一映射表在分支释放时用于精确定位需要减引用的缓存条目，避免通过字符串匹配或遍历查找带来的正确性和性能问题。

**生命周期安全**。引用计数的减操作和物理释放必须原子化执行，避免在并发场景下出现双重释放或泄漏。在遍历缓存池时不得执行删除操作——这是一个常见但容易被忽视的 bug 来源。

### 4.2 Agent PD 调度策略

调度器的设计目标是根据 Agent 任务类型将请求路由到最合适的 Prefill 和 Decode 资源。

**任务分类**。将 Agent 请求分为三类：思考型（THINKING）需要高算力 Prefill 处理长且复杂的 Prompt；交互型（INTERACTIVE）需要低延迟 Decode 实现快速响应；混合型（HYBRID）介于两者之间。

**Worker 选择**。为每个 Worker 维护一组负载指标（当前负载、队列深度、GPU 利用率、平均延迟），通过加权评分函数选择最优 Worker。评分函数需要支持不同偏好维度：

- 思考型 Agent 偏好高剩余算力（`prefer_high_capacity`）
- 交互型 Agent 偏好低延迟（`prefer_low_latency`）
- 混合型走均衡路径

**批量路由的保序性**。批量请求按优先级排序后路由（高优先级先选择资源），但返回结果必须保持与输入请求的对应关系。这要求在排序路由后，将结果按原始索引重新排列。

### 4.3 Hidden State 前缀缓存

前缀缓存的目标是避免对相同图像重复执行 Vision Encoder 计算。

**缓存键的设计**。这是前缀缓存方案中最关键的决策点。理想情况下，缓存键应当精确反映图像内容——相同的图像产生相同的键，不同的图像产生不同的键。但计算精确的图像哈希（如对完整张量做 SHA-256）的开销可能接近甚至超过重新编码的开销，从而使缓存失去意义。

我们采用的方案是采样哈希：从张量中按等间距采样固定数量的元素，与张量的元信息（shape、dtype、device）拼接后计算摘要。这种方法的理论依据是：对于自然图像经过 ViT 编码后的输出，等间距采样在统计意义上能够充分区分不同图像的编码结果。在碰撞概率方面，128 位摘要在千级缓存条目下的碰撞概率可以忽略不计。

**淘汰策略**。采用 LRU（最近最少使用）淘汰，配合可配置的 TTL（存活时间）和容量上限。LRU 的选择依据是多模态推理场景中图像访问通常具有时间局部性——用户在多轮对话中倾向于反复引用最近使用的图像。使用单调时钟（monotonic clock）而非墙上时钟（wall clock）进行时间判断，以避免系统时钟调整对 TTL 计算的影响。

## 5. 技术难点分析

### 5.1 EPD 阶段间的数据格式适配

不同阶段之间的数据格式存在差异。Encoder 输出的是视觉 Hidden States（通常为二维张量 `[num_patches, hidden_dim]`），而 Prefill 期望的输入是 LLM 的 Embedding 格式（`[batch, seq_len, hidden_dim]`）。两者之间需要经过投影（Projection）和拼接（Concatenation）操作。

在 Qwen-VL 等模型中，这一适配更为复杂：Vision Encoder 的输出除了原始 Hidden States，还包含 Deepstack Features——经过不同投影层映射到 LLM 隐藏维度的多层视觉特征，分别注入到 LLM 的不同 Transformer 层中。EPD 分离时，这些 Deepstack Features 需要随 Hidden States 一同传输，并在 Prefill 节点正确注入到对应的 LLM 层。

### 5.2 KV Cache 引用计数的正确性保证

引用计数方案的正确性依赖于严格的计数一致性：每一份物理缓存被多少个活跃分支引用，其引用计数就必须是多少。在实际工程中，以下几类场景容易导致计数不一致：

- **分支释放遗漏**：fork 出的分支如果因为异常或超时未被正常释放，引用计数将永久偏高，导致物理内存无法回收。
- **写时复制中的旧引用清理**：CoW 操作在分配新缓存的同时必须准确减少旧缓存的引用，且在清理过程中不能对正在遍历的数据结构执行删除操作。
- **多级 fork 的传递性**：如果从分支 A fork 出分支 B，再从分支 B fork 出分支 C，释放 B 时不应影响 A 和 C 对物理缓存的访问。

这些问题需要通过严格的映射关系管理（分支到缓存的显式映射）和受控的释放流程（先减引用，再判断是否需要释放物理内存）来解决。

### 5.3 多模态流水线的负载均衡

EPD 三个阶段的计算量和延迟差异较大。以典型的 448×448 图像和 200 token 输入为例，Encoder 约需 30-50ms，Prefill 约需 100ms，Decode 的单 token 延迟约 40-50ms。三个阶段的速度不匹配意味着在稳态下，Decode 节点是瓶颈——每完成一个 token 的 Decode，Prefill 和 Encoder 已经可以处理新的请求。

在生产环境中，需要通过以下方式缓解流水线气泡：

- 为 Encoder 和 Prefill 配置较少的 GPU 资源，为 Decode 配置较多的资源
- 利用 Encoder 输出可缓存的特性，在缓存命中时跳过 Encoder 阶段，直接将 Prefill 作为流水线入口
- 通过调度器将不同类型的请求分配到不同的流水线实例，避免长 Decode 请求阻塞后续的短请求

### 5.4 前缀缓存与 KV Cache 的协同

Hidden State 前缀缓存和 KV Cache 管理的缓存粒度和淘汰策略不同。前者以单张图像的编码结果为缓存单元（通常 1-5 MB），后者以完整请求的 KV Cache 为缓存单元（通常 10-100 MB）。两者需要在 Store 层设计差异化的元数据管理——前者关注内容哈希命中率和 TTL，后者关注引用计数和生命周期。

## 6. 方案取舍

### 6.1 EPD vs PD

EPD 分离引入了额外的 E→P 传输开销，在纯文本场景下没有收益。选择 EPD 的理由是多模态场景中图像输入量大且重复率高，前缀缓存可以显著降低 Encoder 计算开销。当缓存命中率较高时，EPD 分离的净效果是将 Encoder 阶段的计算近乎消除，只保留传输开销。

### 6.2 引用计数 vs 深拷贝

深拷贝方案的优势是实现简单、无共享状态、不存在泄漏风险。劣势是时间和空间代价与分支数线性相关。在 Tree-of-Thought 场景中，典型的分支数为 4-16，深拷贝 30MB 的 KV Cache 8 次意味着 240MB 的额外内存和数毫秒的延迟。引用计数方案虽然实现复杂度高，但克隆操作的时间和空间开销均为 O(1)，在分支数较多时优势明显。

### 6.3 采样哈希 vs 全量哈希

全量哈希保证绝对的内容区分能力，但计算开销较大（涉及完整的 GPU→CPU 数据拷贝和哈希运算），可能使缓存查找的延迟接近甚至超过重新编码的延迟。采样哈希通过牺牲理论上的绝对碰撞安全性换取两个数量级的查找加速。在缓存条目数量在千级以内的场景下，这一取舍在实际工程中是合理的。

### 6.4 TCP vs RDMA

Mooncake Transfer Engine 同时支持 TCP 和 RDMA 传输。在没有 InfiniBand/RoCE 硬件的环境中，TCP 传输仍然可以完成功能验证，但带宽受限于网络接口（通常 10-25 Gbps），对于 P→D 链路的大数据量传输可能成为瓶颈。RDMA 环境下预期可获得 100-200 Gbps 的传输带宽，使跨节点 EPD 分离的传输开销大幅降低。当前方案在传输层做了协议无关的封装，TCP 和 RDMA 模式可以无缝切换。
